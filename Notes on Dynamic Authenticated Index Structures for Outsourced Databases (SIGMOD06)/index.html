Notes on Dynamic Authenticated Index Structures for Outsourced Databases (SIGMOD'06)

Posted on 2022-02-04 | Edited on 2022-02-12 | In Database | Views:

Introduction

  • 3 entities in ODB scenario

    • data owner, server, client.
    • data owner create the database along with the associated index and ADS and uploads to server.

  • 3 dimensions for query authentication:

    • correctness, completeness, freshness (due to potential update)
  • 6 metrics
    • owner’s computation overhead
    • owner-server communication cost
    • server’s storage cost
    • server’s computation cost
    • client-server communication cost
    • client computation cost

  • drawbacks of existing works

    • only consider static scenarios, i.e. owners never issue data updates
    • only consider subset of the 6 metrics above, thus optimized for particular case, not general.
    • performance analyzing is theoretical, doesn’t consider pratical cost of cryptographic primitives?
    • doesn’t conduct empirical evaluation on a working prototype of existing techniques

  • contribution…

  • other security issues in ODB yet orthogonal…

Preliminaries

Any successful scheme must make it computationally infeasible for a malicious server to find incorrect query results and vrification object that will be accepted by a client who has the appropriate authentication information from the owner.

Cryptography essentials

Collision-resistant hash functions

SHA1: random length => 160-bit

Public-key digital signature schemes

1024-bit RSA

signing and verification cost: one hash computation annd one modular exponentiation

Aggregating several signatures

Condensed-RSA

  • multiple signatures signed by the same signer can be combined together and verified once.
  • The combine operation can be done by anyone, no need knowledge of SK.
  • aggregate cost of $t$ signature: $t-1$ modular multiplications
  • verification: $t-1$ multiplications, $t$ hashing operations, one modular exponentiation. (computational gain is that $t-1$ modular exponentiations are replaced by modular multiplications).

The Merkle Hash Tree

Solve the simplest form of the query authentication problem for point queries and datasets that can fit in main memory.

Cost: providing and computing $\log_2n$ hash values, chaper than storing and verifying per data value.

Cost model for SHA1, RSA and Condensed-RSA

hash operation: $2\sim3\mu s$

Modular multiplication, signing, verifying: 50, 1000, 100 times slower than hashing.

Previous work

  • survey: CIKM’05

  • correctness only, basd on MHT: JCS’03, DBSec’00, Algorithmica’04.

  • MHT on DAG: Algorithmica’04
  • promote VB-tree suitable for secondary storage: ICDE’04
  • signature-based, address correctness and completeness: CIKM’05, NDSS’04, SIGMOD’05

The Static Case

  • three approaches for static scenario:
    • signature-based
    • Merkle-tree-like
    • novel embedded tree approach
  • cost model
    • storage
    • construction
    • query
    • authentication

The analysis considers range queries on a speciﬁc database attribute A indexed by a B +-tree, i.e. primary index.

Aggregated Signatures with B+-trees

Direct extension of aggregated signatures and ideas that appeared in [23, 26][23, 26].

  • Owner side
    • owner individually hashes and signs all consecutive pairs of tuples in the db, assuming some sorted order on a given attribut $A$.
      • consecutive pair: $s_i=S(r_i|r_j)$. where $r_i, r_j$ are two consecutive tuples. Chaining tuples in this way enables clients to verify that no in-between tuples have been dropped or modified.
    • First and last tuples can be paired with special marker records.
  • Server side: To speed up, server constructs a B+-tree on top of attribute A.
    • VO: $r_q|s_q$ per query result
    • In addition, one tuple to the left of lower-bound and right of upper-bound is returned to guarantee that no boundary results have been dropped.
  • Two obvious and serious drawbacks:
    • Large VO size: lieaner to # signatures, w.r.t. $N_R$.
    • High verification cost: $N_R$ verification cost for the clients.
    • Solution: aggregated signature signing and verification.

The Merkle B+-tree

An MB-tree works like a B +-tree and also consists of ordinary B + -tree nodes that are extended with one hash value associated with every pointer entry.

After computing all hash values, the owner has to sign the hash of the root using the private key.

Embeded Merkle B-tree

  • Construction
    • Every EMB node consists of regular B+-tree entries, augmented with an embedded MB-tree,
    • EMB tree has fanout $f_e$, EMB node’s embedded MB-tree has fanout $f_k$.
    • Each EMB node stores up to $f_e$ nodes, which are exact the same data items at its embedded MB-tree’s leaf level.
    • Essentially, we are collapsing an MB-tree with height $d_e\cdot d_k=\log_{f_k}N_D$ into a tree with height $d_e$ that stores smaller MB-tree of height $d_k$ within each node.
